## 二、软件需求分析

### 2.1 项目背景与定位

随着环保消费观念普及，二手交易需求持续增长。大型综合平台往往缺乏「同社区信任、自提面交」的本地化体验。本项目定位为**本地化、高信任度的 C2C 二手社区**，通过认证机制降低交易风险，通过 AI 提升商品检索与推荐效率，并以**模拟支付**保障教学与联调的完整闭环。

目标用户为在校学生 / 社区居民（买卖双方）、注重性价比的环保消费者，以及负责内容审核与用户治理的系统管理员。

### 2.2 功能性需求

按一级模块划分：

| 编号 | 模块 | 功能概要 |
|------|------|---------|
| M1 | 用户中心 | 注册登录、个人资料（昵称/头像/联系方式/收货地址）、角色权限 |
| M2 | 商品交易 | 发布、浏览、搜索、编辑、下架、收藏、审核流、热门标签 |
| M3 | 订单支付 | 下单、模拟支付、卖家履约、确认收货、取消、退款 |
| M4 | 评价反馈 | 订单完成评价、卖家信誉聚合、意见反馈工单 |
| M5 | AI 服务 | 智能搜索（向量+关键词）、流式智能客服、个性化推荐、对话日志 |
| M6 | 管理后台 | 商品审核、用户管理、分类管理、订单/退款监管、反馈处理、操作审计 |

**关键业务规则**：

- 新发布或修改核心信息的商品默认进入**待审核**，管理员通过后上架、驳回须填写原因；前台仅展示「已上架」商品。
- 订单状态机：`待付款 → 待履约 → 待收货 → 已完成`，分支含 `已取消`、`退款待审核`、`已退款`。待付款订单可直接取消；已付款走退款申请→管理员审核流程，退款通过后恢复库存。
- 「热门」规则：卖家评分均值 `rating_avg` 非空且 ≥ 4.5 时，其在售商品标记为 `is_hot`。
- AI 搜索失败时**自动降级**为关键词检索，保证前端不白屏。

### 2.3 非功能性需求

- **性能**：首屏目标约 1s，复杂搜索约 2s（含 AI 超时与降级，搜索超时阈值 2s）。
- **安全**：密码 **bcrypt** 加盐哈希，禁止明文存储；JWT 鉴权 + 服务端角色校验防越权；SQLAlchemy ORM 参数化查询防注入；上传图片限定 JPG/PNG/WEBP 白名单、单张 ≤ 2MB，禁止可执行后缀。
- **可用性与兼容**：响应式布局，适配 PC 与移动宽度；异常给出友好提示，不暴露堆栈。
- **可扩展**：支付抽象为可插拔 Provider；REST API 统一 `/api/v1` 版本前缀；数据库与向量检索预留 PostgreSQL + pgvector 演进路径。

## 三、软件设计（概要设计与详细设计）

### 3.1 总体架构

系统采用前后端分离的三层 B/S 架构：

```
┌─────────────────────────────────────────────┐
│  前端 (Vue 3 + TS + Vite)                      │
│  视图层 Views ─ 组件 Components ─ 路由 Router    │
│  状态 Pinia(auth) ─ API 封装(fetch) ─ 401 拦截  │
└───────────────┬─────────────────────────────┘
                │ HTTP / REST (JSON, /api/v1)
                │ Authorization: Bearer <JWT>
┌───────────────▼─────────────────────────────┐
│  后端 (FastAPI)                                │
│  Router (HTTP层) ─ Service (业务层)            │
│  Repository (数据访问层) ─ Schema/Model        │
│  Deps: JWT 鉴权 / 管理员校验                    │
└───────┬───────────────────────┬─────────────┘
        │                       │
┌───────▼────────┐   ┌──────────▼───────────┐
│ PostgreSQL      │   │ 本地 Ollama 大模型      │
│ (SQLAlchemy ORM)│   │ qwen3.5 / mxbai-embed │
│ Alembic 迁移    │   │ Chat / Embedding      │
└─────────────────┘   └──────────────────────┘
        静态资源：uploads/ 经 /static 暴露（商品图、头像）
```

### 3.2 后端分层设计（详细）

后端遵循清晰的分层职责，每层只关心自己的事（参见 `docs/后端架构说明.md`）：

- **入口组装** `app/main.py`：创建 FastAPI 应用，注册 CORS 中间件，统一以 `/api/v1` 前缀挂载各模块路由，挂载 `/static` 静态目录，提供 `/health` 健康检查。
- **依赖注入** `app/deps.py`：`get_current_user`（解析 JWT、校验禁用）、`get_current_admin`（在前者基础上校验 `role == admin`）、`get_optional_user`（可选登录态）。
- **路由层** `app/routers/*`：定义 URL、HTTP 方法、参数来源与鉴权依赖，编排调用下层，返回 Schema。
- **业务层** `app/services/*`：序列化（ORM → 对外结构）、订单状态机/支付/超时、评分聚合、AI embedding 与 RAG 检索等可复用能力。
- **数据访问层** `app/repositories/*`：封装 SQLAlchemy 查询、关联预加载与持久化，向上屏蔽 SQL 细节。
- **数据结构**：ORM 模型 `app/models/*`（对应数据库表）与 Pydantic Schema `app/schemas/*`（对外请求/响应结构）分离。
- **迁移** `alembic/`：维护数据库结构演进；启动时自动升级。

**分层架构设计原则**（判断「逻辑该放哪一层」的依据）：

- 查/写数据库 → **Repository**；业务规则、跨模块流程、计算字段、序列化 → **Service**；URL/鉴权依赖/参数声明/调用编排 → **Router**；对外字段结构 → **Schema**；表结构与关系 → **Model**。
- **关注点分离**：Router 不写 SQL、不含业务细节；Repository 不关心业务、只屏蔽查询细节；Service 承载可复用业务能力（如订单序列化中计算 `payment_expires_at`、从图片推导 `cover_image`），便于在多个路由间复用与单元测试。
- **依赖注入贯穿鉴权**：`Depends` 把数据库会话、当前用户、管理员校验声明式地注入到接口签名上，权限即接口契约的一部分，杜绝「忘记鉴权」。
- **版本化与可演进**：API 统一 `/api/v1` 前缀；支付抽象为 Provider 接口、数据库经 Alembic 迁移、向量存储兼容有/无 pgvector 两种环境，均为后续演进预留空间。

一次典型请求时序（以「买家查看订单详情」为例）：

```
前端 → GET /api/v1/orders/{id}  (Bearer JWT)
  → Router.get_order_detail
      → Depends(get_db)            注入 DB 会话
      → Depends(get_current_user)  解析 JWT、校验未禁用 → user
      → OrderRepository(db).get_detail(id)   预加载 买家/卖家/商品/图片
      → 校验该订单归属当前用户（越权防护）
      → serialize_order_detail(order, user)  Service 组装 OrderDetail
  ← 返回 OrderDetail (Schema, JSON)
```

## 四、软件实现

> 概要设计与详细设计已合并于第三章，此处侧重实现要点与关键技术。

### 4.1 用户与鉴权实现

注册时密码经 **bcrypt** 加盐哈希存库（`core/security.py`），登录校验通过后签发 **JWT**。受保护接口通过 `Depends(get_current_user)` 解析 token、校验账号是否被禁用；管理接口叠加 `get_current_admin` 做服务端角色校验，普通用户访问返回 403，从根本上防止越权。头像上传校验类型（JPG/PNG/WEBP）与大小（≤2MB），文件落地 `uploads/` 并经 `/static` 映射访问。

### 4.2 商品与审核流实现

商品创建/核心信息修改后 `status` 置为待审核；管理员审核通过上架、驳回写入 `reject_reason` 并对卖家可见。列表接口支持分页、分类筛选、关键词搜索与状态过滤，前台仅返回已上架商品。卖家信誉 `rating_avg` 嵌入商品详情/列表，达阈值（≥4.5）的在售商品标记热门。收藏通过 `product_favorites` 关系表实现收藏/取消与「我的收藏」。

### 4.3 订单状态机与模拟支付实现

下单时校验上架状态与库存并扣减，生成「待付款」订单（含单价/金额/收货地址快照）。支付抽象为 `MockPaymentProvider`（`services/payment.py`），预留真实网关接口；`mock-pay` 模拟成功/失败。订单按状态机推进：支付成功→待履约，卖家 `fulfill`→待收货，买家 `confirm-receipt`→已完成。退款走「买家申请→管理员审核」，通过则恢复库存并记审计，驳回回显原因。订单超时未支付由 `order_payment_timeout` 服务处理。

### 4.4 评价与反馈实现

仅「已完成」订单可评价，评价写入后按均值更新卖家 `rating_avg`（`services/rating.py`）。意见反馈作为独立工单，状态流转 `pending → processing → resolved`，管理员可回复处理。

### 4.5 AI 智能服务实现（项目内置 AI 功能）

AI 能力由**本地 Ollama** 提供，聊天模型 `qwen3.5:0.8b`、向量模型 `mxbai-embed-large`（`config.py` 可配置），通过 `httpx` + LangChain 封装在 `services/ai/` 中调用，无需外部云 API，零成本、可离线、数据不外泄。其中 **AI 对话（智能客服）是本项目的核心特色与技术难点**，下文 4.5.1 详述其设计与实现。

- **智能搜索** `POST /ai/search`（`services/ai/search_service.py`）：可选 LLM 抽取查询槽位，对商品向量做余弦相似度召回（top-k），与关键词检索结果融合排序；当向量服务不可用或超时（2s）时**自动降级**为纯关键词检索（`fallback=true`），保证可用性。
- **个性化推荐** `GET /ai/recommendations`（`services/ai/recommend_service.py`）：基于用户浏览历史向量推荐相似商品，数据不足时降级为热门商品。`POST /ai/browse` 记录浏览行为。
- **向量索引**：商品向量以 `embedding_json` 形式存储（兼容无 pgvector 的本机 PG），管理员可 `POST /ai/embeddings/reindex` 重建。

#### 4.5.1 AI 对话（流式智能客服）详细实现

智能客服接口 `POST /ai/chat`，核心实现位于 `services/ai/chat_service.py` 的 `AIChatService`，整体是一套**「检索增强生成（RAG）+ 服务端流式（SSE）+ 多级降级 + 防幻觉兜底」**的对话管线。

**（1）整体处理流程**

```
用户提问 message (+ 可选 conversation_id)
   │
   ▼ _prepare_turn：取/建会话 → 落库 user 消息 → 召回商品 → 构造 prompt
   │   yield meta{status:"retrieving"}
   ▼ _retrieve_products：RAG 检索相关商品
   │   yield meta{status:"generating", product_refs:[...]}  ← 前端先渲染商品卡片
   ▼ OllamaClient.achat_stream：模型流式生成
   │   for chunk: yield chunk{content}                       ← 逐字推送给前端
   ▼ _finalize_assistant_reply：防幻觉校正 → 落库 assistant 消息
       yield done{conversation_id, assistant_message_id, content}
```

整个过程通过 `AsyncIterator[dict]` 产出 `meta / chunk / done / error` 四类事件，路由层 (`routers/ai.py`) 封装为 **SSE 流式响应**逐条下发，前端 `AiChatDrawer` 边接收边逐字渲染，实现「打字机」效果与商品卡片的提前展示。

**（2）会话管理与多轮上下文**

- 传入 `conversation_id` 则续接历史会话并 `touch` 更新时间，否则新建会话并用首句前 40 字自动生成标题。
- 每轮先把用户消息落库 (`ai_messages`)，再取最近 `ai_chat_max_history`（默认 4）条历史拼入 prompt；为控制上下文长度，过长的历史 assistant 回复会截断到 280 字。会话与消息一对多关联，删除会话级联清理消息。

**（3）RAG 检索：召回真实商品作为依据**

`_retrieve_products` 采用「**关键词优先 + 向量可选**」的混合召回策略（由 `ai_chat_keyword_first` / `ai_chat_use_vector` 开关控制）：

- **关键词召回** `_keyword_retrieve`：先用 `expand_search_terms` 对查询做同义扩展（如「苹果手机」→ iPhone 等），逐词在商品库检索去重；若命中为空且识别为「随便看看」类泛意图，则回退取上架商品列表。
- **向量召回** `_vector_retrieve`：将查询经 embedding 模型转向量，在商品向量索引上做相似度检索取 top-k。
- 召回到的商品 id 统一回查、**只保留已上架（`status == "approved"`）**的商品，按 `- 商品名 ¥价格 分类` 格式拼成「相关商品」上下文。

**（4）受约束的 System Prompt（防幻觉）**

为抑制小模型「编造商品」的幻觉，System Prompt 做了强约束：只能依据「相关商品」列表作答；列表有条目时**必须**按列表回答、禁止说「暂无匹配」；仅当列表为「（暂无）」时才可说无货；并显式告知「iPhone 即视为苹果手机的匹配」以对齐口语与库内命名。

**（5）多级降级与兜底，保证「永不白屏」**

- **模型不可用**：`achat_stream` 抛 `OllamaError` 时，不报错中断，而是用召回到的商品名拼出模板回复（「当前找到：…」）并照常落库、产出 `done`。
- **AI 总开关关闭**（`ai_enabled=false`）：直接返回提示语，流程不崩。
- **模型假性否定校正**：`_finalize_assistant_reply` 检测到「明明召回了商品、模型却回答暂无匹配」时，用 `_template_reply_from_products` 以真实商品数据覆盖模型输出，避免小模型答非所问。
- **空回复兜底**：模型输出为空时给出「换个说法或到商品页浏览」的引导。

**（6）线程隔离不阻塞事件循环**

数据库读写（建会话、落库、回查商品）均为同步 SQLAlchemy 调用，通过 `asyncio.to_thread` 包裹，避免阻塞 FastAPI 的异步事件循环，使流式生成与 DB 操作互不拖累。

> 该模块体现了在「**本地小模型 + 真实业务数据**」约束下，如何用 RAG 限定知识边界、用流式提升体验、用多级降级保障可用性、用兜底逻辑修正幻觉，是本项目工程化程度最高的部分。

### 4.6 管理后台实现

`AdminLayout` 侧栏 + 内容区，仅 admin 可达。实现商品审核、用户管理（禁用/重置密码，含审计日志 `admin_audit_logs`）、分类 CRUD、订单与退款监管、反馈处理，所有管理操作均经服务端权限校验与审计记录。

## 六、AI 使用情况说明

本项目的 AI 使用分为两个层面：**（A）开发过程中借助 AI 工具辅助编码**，与**（B）软件产品内置的 AI 功能**。

### 6.1 开发辅助 AI（开发过程中使用的 AI 工具）

**使用的 AI 工具**：在开发过程中使用 AI 编程助手（如 Claude Code / Cursor 等）辅助完成代码编写、调试与文档撰写。

**具体应用场景**：

1. **架构与脚手架搭建**：根据《需求规格说明书》生成前后端工程骨架、分层目录结构与 `/api/v1` 路由分包。
2. **后端模块实现**：辅助编写 SQLAlchemy 模型、Pydantic Schema、Repository 查询与 Router 接口，统一 JWT 鉴权与管理员权限依赖。
3. **订单状态机与退款流程**：协助梳理状态流转、库存扣减/回滚与审计逻辑，定位边界条件。
4. **AI 功能集成**：辅助接入本地 Ollama，实现向量检索、RAG 召回、流式 SSE 客服与失败降级策略。
5. **前端页面与组件**：生成 Vue 视图、路由守卫、Pinia 状态、AI 客服抽屉等组件，并统一响应式样式。
6. **文档撰写**：协助整理《后端架构说明》《实现待办 TODO》与本报告。

**对话流程（典型范式）**：
`描述需求/粘贴报错 → AI 给出方案与代码 → 在工程中应用并运行 → 反馈测试结果或问题 → AI 迭代修正 → 验证通过`。即「需求/问题 → 生成 → 联调 → 反馈 → 修正」的循环。

> 提示：可在此处补充 1–2 段真实对话截图或文字摘录，以满足「对话流程」的具体性要求。

**AI 应用总结**：AI 辅助显著提升了开发效率，尤其在脚手架搭建、样板代码（模型/Schema/接口）生成和报错排查上节省大量时间；同时需要人工把控架构合理性、业务规则正确性与安全细节（如鉴权、越权、输入校验），对 AI 产出进行审阅与测试后再采纳。整体上 AI 作为「结对编程助手」，加速了从需求到可运行系统的落地。

### 6.2 产品内置 AI 功能（软件本身的 AI 能力）

**使用的 AI 工具/模型**：本地 **Ollama** 大模型服务，对话模型 `qwen3.5:0.8b`，向量嵌入模型 `mxbai-embed-large`；通过 `httpx` 与 LangChain 封装在后端 `services/ai/` 中调用，完全本地化、无外部 API 费用。

**具体应用场景与对话流程**：

1. **智能搜索**：用户在商品列表输入自然语言（如「便宜的二手手机」），后端可选用 LLM 抽取查询槽位，再做「向量相似度召回 + 关键词检索」混合排序；服务不可用时自动降级关键词搜索。
2. **智能客服（流式对话）**：用户在 AI 抽屉提问，系统 RAG 召回相关商品作为上下文，受约束的 System Prompt 要求模型仅依据召回列表作答、禁止编造，回答以 SSE 流式逐字返回；会话与消息持久化，支持历史恢复与删除。
3. **个性化推荐**：依据用户浏览历史的向量相似度推荐商品（首页/详情「猜你喜欢」），数据不足时降级为热门商品。

**AI 应用总结**：内置 AI 功能采用**本地化 + RAG + 降级**的稳健设计——以本地 Ollama 避免云 API 成本与隐私外泄，以检索增强（RAG）将回答约束在真实商品数据内、降低幻觉，以「向量失败回退关键词」保证服务始终可用。这套方案在教学/演示环境下兼顾了智能化体验与系统稳定性。
