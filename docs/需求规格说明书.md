# 酱菜交易平台系统需求规格说明书（SRS）

> 版本：1.1  
> 说明：在初版基础上补全用例关系描述，并增加数据模型、API 概要、订单状态、验收标准等可指导实现的内容。

---

## 一、项目概述

### 1.1 项目背景

随着互联网与环保消费观念普及，二手交易需求增长。大型平台往往缺乏本地化（自提、面交、同社区信任）体验。本项目整合 **Vue 前端** 与 **Python 后端**，提供高效、安全、便捷的本地二手买卖服务。

### 1.2 目标用户

- 在校学生 / 社区居民：主要买卖双方。  
- 极简主义 / 环保人士：偏好高性价比二手。  
- 系统管理员：内容审核与用户治理。

### 1.3 产品定位

- **定位**：本地化、高信任度 C2C 二手社区。  
- **价值**：实名/认证降低风险；AI 提升检索与推荐；**模拟支付**保障教学与联调闭环。

---

## 二、总体目标与功能范围

### 2.1 总体目标

用户与商品全流程数字化；AI 客服（问答、意图搜索、行为推荐）；审核机制；模拟支付并可扩展真实网关。

### 2.2 一级模块

| 编号 | 模块 | 概要 |
|------|------|------|
| M1 | 用户中心 | 注册登录、个人信息、角色权限 |
| M2 | 商品交易 | 发布、浏览、搜索、编辑下架、热门标签 |
| M3 | 订单支付 | 下单、模拟支付、状态、取消/确认收货 |
| M4 | 评价反馈 | 交易评价、意见反馈 |
| M5 | AI 服务 | 智能问答、自然语言搜索、推荐、对话日志 |
| M6 | 管理后台 | 商品审核、用户管理、分类管理、订单/退款监管 |

## 三、详细功能需求

### 3.1 用户管理

- 用户名 + 密码注册；登录签发 **JWT**（或 Session）；密码加盐哈希（见 5.2）。  
- 个人信息：昵称、头像、联系方式、收货地址。  
- 角色：普通用户、管理员；管理接口必须服务端校验角色。  
- **实名（可选）**：一期可定义为补充资料 + 管理员标记「已认证」。

### 3.2 商品管理

- 发布：名称、描述、价格、多图、分类、成色、交易方式（自提/邮寄）、库存（二手常见为 1）。  
- 展示：轮播、卖家信誉摘要、参数。  
- 编辑/下架；新商品默认**待审核**，通过后上架，驳回需原因。  
- **热门**：规则须在详细设计中固定（例如：卖家近 N 条评价均分不低于 4.5，且商品在售）。

### 3.3 交易与支付

- 下单：待付款；备注；校验上架与库存。  
- **模拟支付**：成功后进入「待发货/待自提」（或统一为「待履约」）。  
- 确认收货 → 已完成；取消：待付款可直接取消；已付款走退款流程。  
- 并发：下单扣减/锁定库存，与取消、退款回滚策略在详细设计中写清。

**建议状态集合**：待付款 → 待履约 → 已完成；分支：已取消、退款待审核、已退款、退款失败。

### 3.4 AI 客服

- 自然语言 → 意图/槽位 → 查询；失败则降级关键词搜索。  
- 基于浏览历史的推荐。  
- 持久化对话日志（注意隐私与脱敏）。

### 3.5 管理员后台

- 商品审核；用户禁用与密码重置（需审计）；分类 CRUD；退款审核。

### 3.6 评价与反馈

- 订单完成后评价；意见反馈独立工单流。

### 4.2 订单状态机（文字）

1. 创建订单：待付款。  
2. 模拟支付成功：待履约（UI 可区分发货/自提）。  
3. 用户取消（仅待付款）：已取消。  
4. 确认收货：已完成。  
5. 申请退款（符合条件）：退款待审核 → 管理员通过则已退款并恢复库存；拒绝则退款失败，订单回到待履约或按业务规则固定。

## 五、非功能性需求

### 5.1 性能

首屏目标约 1s 内（可测指标）；复杂搜索约 2s 内（含 AI 超时与降级）。初期支持约 500 并发在线；数据量目标：商品约 10 万条、用户约 1 万条。

### 5.2 安全

- 密码：**bcrypt** 或 **Argon2**；禁止明文；不推荐单独 SHA-256 无盐。  
- 鉴权与越权防护；SQL 注入防护（ORM/参数化）；XSS 与 CSP；Cookie 场景考虑 CSRF。  
- 上传文件类型/大小白名单，禁止可执行后缀。

### 5.3 可用性与兼容

响应式布局；Chrome、Edge、Firefox 最近两个主版本；异常时友好提示，不暴露堆栈。

### 5.4 可扩展

支付抽象为可插拔 Provider；REST API 建议 `/api/v1` 版本前缀。

### 5.5 运维（补充）

结构化日志、健康检查、备份策略同原文第六章建议。

## 六、数据与存储

- UGC：用户、商品、订单、评价、反馈。  
- 系统：AI 日志、审计日志。  
- 数据库：规格推荐 **SQL Server**；开发可用 **SQLite**，通过配置切换。  
- 图片：本地或对象存储，库中存路径。

### 核心实体概要

User、Category、Product、ProductImage、Order、Review、Feedback、AIConversation（字段表见后续《数据库设计说明书》）。

## 七、REST API 概要

前缀 `/api/v1`，登录接口携带 `Authorization: Bearer <token>`。

示例：`POST /auth/register`、`POST /auth/login`、`GET/PATCH /users/me`、`GET/POST/PATCH /products`、`POST /orders`、`POST /orders/{id}/mock-pay`、`POST /orders/{id}/confirm-receipt`、`POST /orders/{id}/cancel`、`POST /orders/{id}/refund-request`、`POST /ai/search`、`POST /ai/chat`，以及 `/admin/*` 审核与管理端点。详细请求体与错误码在《接口设计说明书》定义。

---

## 八、验收标准（节选）

- 注册登录与错误提示正确；发布后待审核，驳回可见原因，通过后可买。  
- 库存与订单状态正确；模拟支付与确认收货闭环。  
- 普通用户访问管理接口返回 403。  
- AI 搜索失败可降级，前端不白屏。  
- 主要页面 PC 与移动宽度下布局可用。
